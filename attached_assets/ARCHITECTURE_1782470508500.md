## 3. Process Model

Electron runs two separate OS processes that communicate only through IPC:

```
┌─────────────────────────────────────────────────────────────┐
│  MAIN PROCESS  (Node.js — full OS access)                   │
│                                                             │
│  electron/main/index.ts   ← window lifecycle, IPC routing  │
│  electron/main/downloader.ts  ← download engine            │
│                                                             │
│  • Opens BrowserWindows to render manga pages               │
│  • Downloads images via net.fetch (bypasses CORS)           │
│  • Writes .cbz files to disk                                │
│  • Manages cancel flags per job                             │
└────────────────┬────────────────────────────────────────────┘
                 │  ipcMain.handle / webContents.send
                 │  (serialised JSON — no shared memory)
┌────────────────▼────────────────────────────────────────────┐
│  PRELOAD SCRIPT  (Node context + DOM access)                │
│                                                             │
│  electron/preload/index.ts                                  │
│  • contextBridge.exposeInMainWorld('api', { … })            │
│  • Translates renderer calls → ipcRenderer.invoke          │
│  • Translates ipcRenderer.on events → callback API          │
└────────────────┬────────────────────────────────────────────┘
                 │  window.api.*
┌────────────────▼────────────────────────────────────────────┐
│  RENDERER  (Chromium — no Node access)                      │
│                                                             │
│  src/App.tsx + src/components/JobCard.tsx                   │
│  • Pure React UI                                            │
│  • Calls window.api.startJob / cancelJob / pickFolder       │
│  • Receives JobEvents via window.api.onJobEvent callback    │
└─────────────────────────────────────────────────────────────┘
```

**Security model:** `contextIsolation: true` (default) means renderer JS cannot
access Node APIs directly. Everything goes through the preload bridge.

## 4. Data Flow — End to End

```
User clicks "Start"
        │
        ▼
App.tsx: window.api.startJob(opts)
        │
        ▼  ipcRenderer.invoke('start-job', opts)
        │
        ▼
index.ts: ipcMain.handle('start-job')
  • generates jobId
  • calls runJob(jobId, opts, isCancelled) [fire & forget]
  • returns jobId immediately → renderer adds job card
        │
        ▼
runJob() ─── Phase 1: discoverChapters()
              │  For each chapter up to `count`:
              │  1. loadPage(win, url)          — Electron BrowserWindow
              │  2. getPageTitle(win)            — executeJavaScript
              │  3. findChapterLink(win, dir)    — executeJavaScript
              │  Emits: 'discover' event per chapter
              │
              ├── Phase 2: runParallel()
              │   Up to CFG.CONCURRENCY chapters at once:
              │   downloadChapter() for each
              │     1. createHiddenWindow()
              │     2. loadPage(win, url)
              │     3. scrollAndCollect(win)     — scroll + harvest img srcs
              │     4. downloadImage() × N       — net.fetch per image
              │     5. packageCBZ()              — JSZip → .cbz on disk
              │   Emits: 'chapter' + 'progress' events
              │
              └── Emits: 'done' event
                        │
                        ▼  webContents.send('job-event', event)
                        │  ipcRenderer.on('job-event', …)
                        ▼
              App.tsx: setJobs() — updates React state
                        │
                        ▼
              JobCard re-renders with new status
```

## 5. Shared Types (`src/types.ts`)

Imported by **both** main process and renderer. Never import renderer-only or
Node-only types here.

### `Direction`
```ts
type Direction = 'next' | 'prev'
```
Controls which chapter link (`findChapterLink`) is followed during discovery.

### `Phase`
```ts
type Phase = 'discovering' | 'downloading' | 'done' | 'error'
```
Top-level job phase. Used in `JobEvent` and renderer `Job` model.

### `ChapterStatus`
```ts
type ChapterStatus =
  | 'pending'    // queued, not started
  | 'loading'    // BrowserWindow loading the URL
  | 'scrolling'  // auto-scrolling to trigger lazy-loaders
  | 'fetching'   // downloading individual images
  | 'packaging'  // creating the .cbz zip
  | 'done'       // .cbz written successfully
  | 'error'      // failed — see Chapter.error
```

### `Chapter`
```ts
interface Chapter {
  url:         string          // source chapter URL
  title:       string          // extracted from <h1> or document.title
  status:      ChapterStatus
  fetchDone?:  number          // images downloaded so far
  fetchTotal?: number          // total images on page
  pages?:      number          // pages written to .cbz
  file?:       string          // absolute path to .cbz on disk
  error?:      string          // error message if status === 'error'
}
```

### `StartJobOpts`
```ts
interface StartJobOpts {
  startUrl:   string      // URL of the first chapter to process
  direction:  Direction   // follow 'next' or 'prev' chapter links
  count:      number      // how many chapters to process (≥1)
  outputDir:  string      // absolute path chosen by user (dialog)
}
```

### `JobEvent` (discriminated union)
Every event carries `jobId` so the renderer can route it to the right job card.

| `type` | Extra fields | Meaning |
|---|---|---|
| `'phase'` | `phase`, `chapters?` | Job entered a new phase. `chapters` is set when transitioning to `'downloading'` — pre-populates the chapter list |
| `'discover'` | `current`, `total`, `url` | One chapter URL found during discovery |
| `'chapter'` | `chapter: Chapter` | Full chapter state update (any status transition) |
| `'progress'` | `done`, `total` | N of M chapters finished |
| `'done'` | `message`, `outputDir` | Job complete (success or cancelled) |
| `'error'` | `message` | Job-level error (discovery failed, etc.) |

## 6. IPC API Contract

All channels use `ipcMain.handle` (request/response) except `'job-event'` which
is a push from main → renderer via `webContents.send`.

### `'pick-folder'` → `string | null`
Opens the OS folder-picker dialog. Returns selected path, or `null` if cancelled.

```ts
ipcMain.handle('pick-folder', async () => {
  const { filePaths } = await dialog.showOpenDialog(mainWindow, {
    properties: ['openDirectory', 'createDirectory'],
  })
  return filePaths[0] ?? null
})
```

### `'open-folder'` (dir: string) → `void`
Opens a directory in the OS file explorer via `shell.openPath`.

### `'cancel-job'` (jobId: string) → `void`
Sets `cancelFlags.get(jobId) = true`. The running job checks `isCancelled()` at
every async boundary and stops cleanly.

### `'start-job'` (opts: StartJobOpts) → `string` (jobId)
Generates a unique `jobId = 'job_' + Date.now()`, spawns `runJob()` as a
fire-and-forget async task, and returns `jobId` immediately so the renderer can
add a card before any work begins.

### `'job-event'` ← push (event: JobEvent)
Not invoked by renderer. Main process pushes updates via:
```ts
mainWindow.webContents.send('job-event', event)
```
Preload forwards these through `ipcRenderer.on('job-event', …)` to the
renderer's `onJobEvent` callback.

## 7. Main Process (`electron/main/index.ts`)

### `createMainWindow(): void`
Creates the single `BrowserWindow` with:
- Size: 860×680, minimum 700×500
- `titleBarStyle: 'hiddenInset'` — native traffic-light buttons on macOS
- `contextIsolation: true` — renderer cannot access Node
- Preload: `out/preload/index.js` (compiled from `electron/preload/index.ts`)
- Dev: loads `ELECTRON_RENDERER_URL` (electron-vite HMR server)
- Prod: loads `out/renderer/index.html` from disk

### `emit(event: JobEvent): void`
Safe broadcast to the renderer. Guards against destroyed `webContents`
(can happen if user closes window mid-download).

### `cancelFlags: Map<string, boolean>`
Per-job cancel state. Key = `jobId`, value = `true` when cancelled.
Cleaned up with `cancelFlags.delete(jobId)` when the job finishes normally.

### `runJob(jobId, opts, isCancelled): Promise<void>`
Top-level job orchestrator. Sequence:
1. Emit `{ type: 'phase', phase: 'discovering' }`
2. Call `discoverChapters()` → get chapter list
3. Guard: exit early if 0 chapters found or cancelled
4. Emit `{ type: 'phase', phase: 'downloading', chapters }`
5. Call `runParallel()` → download all chapters
6. Emit `{ type: 'done', message, outputDir }`

Error handling: any uncaught throw from `runJob` is caught at the call site
and emits `{ type: 'error', message }`.

## 8. Downloader (`electron/main/downloader.ts`)

The entire download engine. All functions are pure (no side effects except
the ones documented). Import them individually for unit testing.

### `CFG` — Tuning constants
```ts
export const CFG = {
  SCROLL_STEP_PX:  2000,   // px scrolled per step
  SCROLL_DELAY_MS:  400,   // ms between scroll steps
  STABLE_NEED:       10,   // consecutive same-position reads → bottom reached
  PAGE_TIMEOUT_MS: 20_000, // ms before loadPage times out
  CONCURRENCY:        3,   // max parallel chapter downloads
}
```
Change these at the top of the file — no other code needs editing.

---

### Window Helpers

#### `createHiddenWindow(): BrowserWindow`
Creates an off-screen `BrowserWindow` with `show: false` and `contextIsolation: false`.
`contextIsolation: false` is required so `executeJavaScript()` can read DOM state
without the security sandbox blocking it. These windows are never shown to the user.

**Lifecycle:** Always created at the start of `downloadChapter()` and
`discoverChapters()`, always destroyed in their `finally` block.

#### `sleep(ms): Promise<void>`
Thin wrapper over `setTimeout`. Used to give lazy-loaders time to react after
each scroll step.

#### `loadPage(win, url): Promise<void>`
Loads a URL into a `BrowserWindow` and waits for `'did-finish-load'` or
`'did-fail-load'`. Rejects after `CFG.PAGE_TIMEOUT_MS` milliseconds.

---

### Scroll & Harvest

#### `scrollAndCollect(win): Promise<ImageRef[]>`
Core scraping function. Algorithm:
1. `window.scrollTo(0, 0)` — reset to top
2. Wait 600 ms for initial images
3. Loop: `scrollBy(0, SCROLL_STEP_PX)` → wait `SCROLL_DELAY_MS` → read `scrollTop`
4. If `scrollTop` hasn't changed for `STABLE_NEED` consecutive reads → bottom reached
5. Execute JS to collect all `<img>` elements, extracting `src` from these
   attributes in priority order:
   - `data-src`
   - `data-lazy-src`
   - `data-original`
   - `data-url`
   - `srcset` (first entry)
   - `src`
6. Filter: skip non-http URLs, duplicates, and images ≤150×150 px (icons/avatars)
7. Return sorted by vertical position (`getBoundingClientRect().top + scrollY`)

**Reader container selectors** (tried in order, falls back to `document`):
```
main.rpage-main, .rpage-long-strip, #readerarea, .reader-area,
.wp-manga-chapter-img, .read-container, .manga-container,
.entry-content, main, article
```

#### `ImageRef`
```ts
interface ImageRef {
  src: string   // absolute URL
  top: number   // vertical pixel position (for sort order)
}
```

---

### Navigation

#### `findChapterLink(win, direction): Promise<string | null>`
Finds the next or previous chapter URL by trying two strategies:

**Strategy 1 — CSS selectors** (tried in order):

Next:
```
a.next-chapter, a.next_chapter, a.nextchap, [class*="next"] a,
a[rel="next"], .btn-next a, .chapter-next a, a[aria-label*="next" i]
```

Prev:
```
a.prev-chapter, a.prev_chapter, a.prevchap, [class*="prev"] a,
a[rel="prev"], .btn-prev a, .chapter-prev a, a[aria-label*="prev" i]
```

**Strategy 2 — Text fallback** (searches `nav`, `header`, `footer`, `[class*="nav"]`,
`[class*="chapter"]` first, then all `<a>` tags):

Matches link text + className + aria-label against:
- Next: `/\bnext\b|siguiente|prochain|weiter/i`
- Prev: `/\bprev(ious)?\b|anterior|précédent|zurück/i`

Returns `null` if no match. Discovery stops early when `null` is returned.

#### `getPageTitle(win): Promise<string>`
Reads `document.querySelector('h1')?.innerText ?? document.title ?? 'Manga'`.
Used as the chapter title and to derive the series folder name.

---

### Filename Helpers

#### `cleanTitle(raw): string`
Strips filesystem-illegal characters (`\ / : * ? " < > |`), collapses whitespace
to underscores, trims, and caps at 100 characters.

#### `extractSeries(title): string`
Extracts the series name from a chapter title like `"One Piece - Chapter 1083"`.
Regex: `/^(.+?)\s*[-–—]?\s*(?:Chapter|Chap|Ch\.?)\s*[\d.]/i`

Falls back to `cleanTitle(title)` if no match (used as the output subfolder name).

---

### Image Download

#### `downloadImage(url, referer): Promise<Buffer>`
Downloads one image using `electron.net.fetch` (Electron's Node-layer HTTP client).
- Sets `Referer` header to the chapter page URL (required by many CDNs)
- Sets a realistic `User-Agent` to avoid bot-blocks
- Throws on non-2xx status

#### `guessExt(buf): string`
Detects image format from magic bytes:
| Magic bytes | Format |
|---|---|
| `FF D8` | jpg |
| `89 50` | png |
| `52 49` | webp |
| `47 49` | gif |
| (fallback) | jpg |

---

### CBZ Packaging

#### `packageCBZ(pages, title, series, outputDir): Promise<string>`
1. Creates a JSZip instance
2. Adds each page as `001.jpg`, `002.png`, etc. (zero-padded, extension from `guessExt`)
3. Generates a Node buffer with `compression: 'STORE'` (no compression — comic
   readers expect uncompressed CBZ for fast random access)
4. Creates `outputDir/series/` with `mkdirSync({ recursive: true })`
5. Writes `outputDir/series/cleanTitle(title).cbz`
6. Returns the absolute file path

---

### Chapter Download

#### `downloadChapter(chapter, outputDir, onUpdate, isCancelled): Promise<void>`
Full lifecycle for one chapter. All status transitions are reported via `onUpdate`:

```
pending → loading → scrolling → fetching (0/N … N/N) → packaging → done
                                                                  ↘ error
```

`onUpdate` accepts a `Partial<Chapter>` merged with defaults, so callers always
get a complete `Chapter` object even if only one field changed.

Images that fail to download individually are skipped with a `console.warn` —
the chapter still packages as long as at least one image succeeded.

Cancellation is checked before each image download. When cancelled, throws
`'Cancelled'` which is caught and recorded as `status: 'error'`.

The hidden `BrowserWindow` is always destroyed in `finally`, even on error.

---

### Discovery

#### `discoverChapters(startUrl, direction, count, onDiscover, isCancelled)`
Phase 1 of a job. Uses a **single** hidden window (reused across chapters for
efficiency). For each chapter:
1. Calls `onDiscover(i+1, count, url)` — lets renderer show discovery progress
2. Loads the page
3. Reads title
4. Follows the next/prev link for the next iteration

Returns early if `findChapterLink` returns `null` (end of series reached).
Returns `Pick<Chapter, 'url' | 'title'>[]` — the full `Chapter` objects are
built during Phase 2.

---

### Parallel Queue

#### `runParallel(chapters, outputDir, onChapter, onProgress, isCancelled)`
Processes up to `CFG.CONCURRENCY` chapters simultaneously using a
promise-based sliding-window queue (no external library).

Algorithm:
```
active = Set of in-flight chapter URLs
queue  = [...chapters]

next():
  while active.size < CONCURRENCY and queue not empty:
    pop chapter from queue
    add to active
    downloadChapter().then(() => {
      remove from active
      done++
      onProgress(done, total)
      call next() again
    })
  if queue empty and active empty → resolve
```

`onProgress` fires after each chapter completes, regardless of success/failure.

## 9. Preload Bridge (`electron/preload/index.ts`)

Exposes exactly 5 methods on `window.api` via `contextBridge`:

```ts
contextBridge.exposeInMainWorld('api', {
  startJob:   (opts) => ipcRenderer.invoke('start-job', opts),   // → string (jobId)
  cancelJob:  (id)   => ipcRenderer.invoke('cancel-job', id),    // → void
  pickFolder: ()     => ipcRenderer.invoke('pick-folder'),        // → string | null
  openFolder: (dir)  => ipcRenderer.invoke('open-folder', dir),  // → void
  onJobEvent: (cb)   => ipcRenderer.on('job-event', (_e, data) => cb(data)),
})
```

`onJobEvent` is a subscription (not request/response). The callback is called
every time main emits a `'job-event'`. There is intentionally no unsubscribe —
the listener lives for the app's lifetime.

## 11. Renderer — App (`src/App.tsx`)

### Renderer-side `Job` model
```ts
interface Job {
  id:               string
  startUrl:         string
  direction:        Direction
  phase:            'discovering' | 'downloading' | 'done' | 'error'
  discoverCurrent?: number     // chapters found so far
  discoverTotal?:   number     // total chapters to find
  chapters:         Map<string, Chapter>   // keyed by URL
  chapterOrder:     string[]               // insertion-order URLs
  done:             number     // chapters fully completed
  total:            number     // total chapters in job
  message?:         string     // final message on done/error
  outputDir:        string
}
```

`chapters` is a `Map` (not array) so individual chapter updates are O(1).
`chapterOrder` is a separate array to preserve insertion order for rendering.

### State
```ts
startUrl:   string       // controlled input
direction:  Direction    // 'next' | 'prev' toggle
count:      number       // chapter count, min 1
outputDir:  string       // from dialog — displayed read-only
jobs:       Map<string, Job>   // keyed by jobId
```

### `useEffect` — event listener
Registered once on mount. Handles all 6 `JobEvent` types and updates the
relevant `Job` in the `jobs` map immutably (`new Map(prev)`).

Each `Job` is spread on update (`{ ...job }`) so React detects the reference
change and re-renders the affected `JobCard`.

### `startJob`
Validates inputs, calls `window.api.startJob(opts)`, receives `jobId`, and
inserts a new `Job` into state immediately (before any events arrive) so the
card appears instantly.

### `cancelJob`
Calls `window.api.cancelJob(jobId)`. The cancel is async — the card stays
visible and transitions to `done` or `error` when the main process acknowledges.

### `jobList`
`Array.from(jobs.values()).reverse()` — newest job at the top.

## 14. Build Config (`electron.vite.config.ts`)

```
main     → input: electron/main/index.ts
             plugin: externalizeDepsPlugin()   ← keeps Node built-ins external
             output: out/main/index.js

preload  → input: electron/preload/index.ts
             plugin: externalizeDepsPlugin()
             output: out/preload/index.js

renderer → root: src/
             input: src/index.html
             plugin: @vitejs/plugin-react
             output: out/renderer/
```

`externalizeDepsPlugin()` marks all `node_modules` as external in the main/preload
bundles. They rely on Node's `require()` at runtime, not bundled copies.

In dev mode, the renderer runs as a Vite HMR server (`localhost:5173`). The main
process loads it via `ELECTRON_RENDERER_URL`. In production, the renderer is
built to static files loaded with `win.loadFile(...)`.

## 17. Tuning Reference

All in `CFG` at top of `electron/main/downloader.ts`:

| Constant | Default | Effect of increasing |
|---|---|---|
| `SCROLL_STEP_PX` | 2000 | Faster scroll, may miss images |
| `SCROLL_DELAY_MS` | 400 | More time for lazy-loaders to fire |
| `STABLE_NEED` | 10 | More confirmation reads before declaring bottom |
| `PAGE_TIMEOUT_MS` | 20 000 | More patience for slow sites |
| `CONCURRENCY` | 3 | More parallel downloads (higher memory + CPU) |

**Slow sites / missed images:** increase `SCROLL_DELAY_MS` to 600–800 and
`STABLE_NEED` to 15.

**Fast, well-structured sites:** `SCROLL_DELAY_MS: 200`, `STABLE_NEED: 5`.

## 18. Extension Guide

### Add a new IPC channel

1. `electron/main/index.ts` — add `ipcMain.handle('channel-name', handler)`
2. `electron/preload/index.ts` — add to `contextBridge.exposeInMainWorld('api', { … })`
3. `src/env.d.ts` — add the method signature to `Window['api']`
4. `src/App.tsx` (or component) — call `window.api.channelName()`

### Support a new manga site's next/prev buttons

Edit `findChapterLink()` in `downloader.ts`. Add the site's CSS selector to the
appropriate `selectors` array. If the site uses JavaScript navigation (no `<a href>`),
add a special-case branch before the selector loop.

### Support series index pages (scan all chapters at once)

1. Add a new function `scrapeSeriesIndex(url): Promise<string[]>` in `downloader.ts`
   that loads the series page and collects all chapter URLs.
2. Add a new IPC channel `'start-series-job'` in `main/index.ts` that calls
   `scrapeSeriesIndex` instead of `discoverChapters`.
3. Add a "Series URL" input field in `App.tsx`.

### Change the output filename format

Edit `cleanTitle()` and/or `extractSeries()` in `downloader.ts`.
The CBZ path is: `outputDir / extractSeries(title) / cleanTitle(title).cbz`

### Skip already-downloaded chapters

In `downloadChapter()`, before `createHiddenWindow()`:
```ts
import { existsSync } from 'fs'
const expectedFile = join(outputDir, extractSeries(title), `${cleanTitle(title)}.cbz`)
if (existsSync(expectedFile)) {
  emit({ status: 'done', title, pages: 0, file: expectedFile })
  return
}
```

### Add a retry mechanism for failed images

Replace the single `downloadImage()` call in `downloadChapter()` with a retry loop:
```ts
async function fetchWithRetry(url: string, referer: string, retries = 3): Promise<Buffer> {
  for (let i = 0; i < retries; i++) {
    try { return await downloadImage(url, referer) } catch (e) {
      if (i === retries - 1) throw e
      await sleep(1000 * (i + 1))
    }
  }
  throw new Error('unreachable')
}
```

## 19. Known Limitations

| Limitation | Cause | Workaround |
|---|---|---|
| Sites with Cloudflare challenge fail | CF blocks headless Electron | None currently — CF-protected sites need a logged-in session cookie |
| Infinite-scroll sites may not fully collect | `STABLE_NEED` reached before all images loaded | Increase `SCROLL_DELAY_MS` and `STABLE_NEED` |
| `next/prev` detection fails on some sites | Non-standard navigation markup | Add site-specific selector to `findChapterLink()` |
| Single hidden window per discovery | Reused window state can carry cookies/cache across chapters | Intentional — helps with session continuity |
| No deduplication across jobs | Same chapter can be downloaded twice in two jobs | Add `existsSync` check (see Extension Guide) |
| Cancel is not instant | `isCancelled()` is polled, not signalled | Cancels cleanly at the next await point |
| `@types/jszip` is a stub | jszip ships its own types | Remove `@types/jszip` from devDependencies safely |
